# Sharded HAMT Copy-on-Write (CoW) Map

A highly concurrent, thread-safe, and lock-free dictionary implementation in Go
1.23+ leveraging a Sharded Hash Array Mapped Trie (HAMT) architecture combined
with Read-Copy-Update (RCU) path-copying semantics.

This structure is tailored for maximum read performance and minimized CPU
cache-line bouncing under high concurrent write contention.

This repo actually contains many implementations (all generated by AI) and this
was the implementation history

1. b-tree based
1. to multi-element tree version, 60% faster (than b-tree version)
1. hamt version, 65% faster (faster in read heavy, slower in write heavy workload than multi element tree version)
1. hamt sharded version, 80% faster (less contention on write on root element, overall faster than multi-element tree version)

### 1. Contention Sharding
To circumvent the bottleneck of a single global CAS loop, the map pre-allocates **64 independent shards**. Incoming keys are routed to a targeted shard based on the lower bits of their hash value:
$$\text{Shard Index} = \text{Hash} \pmod{64}$$
This mapping divides thread contention by up to 98% on multi-core systems, isolating atomic compare-and-swap operations to separate hardware execution pipelines.

### 2. Hash Array Mapped Trie (HAMT)
Inside each shard, entries are arranged within a sparse prefix-trie.
* **Branching Factor:** Each node features a 32-way branching layout ($W=32$).
* **Bitmapped Populating:** Instead of a bloated 32-slot pointer array, nodes use a single `uint32` bitmap coupled with a packed, variable-length storage slice. The exact offset of an active index is determined via bitwise population counting:
  $$\text{Index Position} = \text{bits.OnesCount32}(\text{Bitmap} \ \& \ (\text{Mask} - 1))$$
* **Shallow Traversal Depth:** Traversal uses a 5-bit window per tier. For a standard 32-bit hash space, the structure guarantees a maximum depth of 7 hops.

### 3. Read-Copy-Update (RCU) & Path-Copying
* **Zero-Lock Reads:** Readers load the shard's root element instantly using `atomic.Pointer.Load()`. Traversal down the trie is purely non-blocking, zero-allocation, and free from memory fences.
* **Point-in-Time Iterators:** Standard library `iter.Seq2` loops capture a root snapshot reference at invocation. Concurrent structural mutations remain entirely invisible to that active iteration stream, eliminating data races without keeping long-lived lock assertions.
* **Isolated Mutations:** Modification logic duplicates specific path segments downward to the targeting node. If a competing thread updates the root pointer mid-flight, the current retry loop discards the uncommitted branch clone, guaranteeing that out-of-bounds corruption never leaks across competing threads.

### 4. Hash Collision Fallback
If two distinct keys generate identical 32-bit hashes, the trie caps its downward branch expansion at a maximum threshold. It transitions the collided bucket into an explicit list node, falling back to thin linear scanning to avoid stack overflows.


## Performance Considerations

* **When to use:** Ideal for read-heavy workloads, configuration registries, routing dictionaries, and low-to-medium writer pools where snapshot isolation guarantees are required.
* **Memory profile:** Shifts node memory recovery onto the Go standard Runtime Garbage Collector. Allocation footprints adapt based on update frequencies and total key density distribution.
